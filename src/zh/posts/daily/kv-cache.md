# KV Cache

## 一、基础定义与作用

KV Cache（Key-Value Cache，键值缓存） 是大模型（Transformer 架构）推理时的核心加速技术，通过缓存历史 token 的 Key 和 Value 向量，避免重复计算，把自回归生成的时间复杂度从 O(n²) 降到 O(n)，大幅提速。

仅记住定义约等于基础水平；下面三个问题是面试官常深挖的方向。

## 二、核心问题一：为什么缓存 K 和 V，而不是 Q？

自回归生成中，每生成一个新 token，模型需要用**当前 token 的 Query**，与**所有历史 token 的 Key** 做点积得到注意力权重，再对**历史 token 的 Value** 加权求和。

| 向量 | 特点 |
|------|------|
| **Query（Q）** | 只对应当前新 token，用完即弃，无需长期保留 |
| **Key / Value（K、V）** | 对应所有**历史** token，随生成长度不断累积 |

若不缓存，每生成一个新 token 都要**重新计算**所有历史位置的 K 和 V（例如第 100 个 token 要重算前 99 个），重复计算量巨大。

**结论**：KV Cache 缓存的是**累积的 K、V**，一般不缓存一次性使用的 Q。

## 三、核心问题二：KV Cache 的代价

KV Cache 是**用显存换算力**，不是「免费加速」。

### 估算示例

假设：**40 层**、**32 头**、**头维度 128**、**上下文长度 4096**、**fp16** 存储。

粗略规模：

```
层数 × 头数 × 序列长度 × 头维 × (K+V 两份) × sizeof(fp16)
= 40 × 32 × 4096 × 128 × 2 × 2 字节
≈ 2.5 GB（单请求）
```

若同时服务 **64** 个用户，仅 KV 相关显存约 **160 GB**，可能超过单卡 A100（80 GB），因此 **KV Cache 体积直接限制并发与上下文长度**。

## 四、核心问题三：为什么 KV Cache 是推理核心瓶颈？

大模型推理通常分两阶段：

| 阶段 | 含义 | 瓶颈倾向 |
|------|------|----------|
| **Prefill** | 并行处理用户已输入的 token，建立初始 KV | **算力**（GPU 越强越快） |
| **Decode** | 每步只生成 1 个 token，需读**整段** KV 与当前 Q 做注意力 | **显存带宽**：计算量相对小，但数据搬运大，算力利用率往往不高 |

Decode 阶段反复读写 KV，因此 **KV 的布局、分页、精度**（如 PagedAttention、共享 KV、fp8/int4）成了推理系统优化的主战场。

## 五、小结

1. **只存 K、V**：历史复用、Q 一次性。  
2. **显存昂贵**：长上下文 × 并发直接撑爆显存。  
3. **Decode 偏带宽**：优化 KV 占用与访问模式以提升吞吐。
